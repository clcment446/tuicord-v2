# Code-base Cleanup Report — tuicord-v2 — 2026-07-17

## Verdict

The core terminal toolkit has good regression coverage for width, screen diffs, and chat-cache equivalence, and the focused UI/markup/TUI suite plus `go vet` are clean. The most urgent issue is not file size by itself: `chatLine` carries render text together with interactive and loading metadata, but its frame/prefix transforms copy only some fields. That produces already-shipped rich-content interaction regressions. An architectural rewrite is not warranted; introduce one lossless line-transform helper, repair the direct regressions with tests, then split the large chat renderer along existing responsibilities.

Audit scope: all tracked production Go source was inventoried; detailed review prioritized the 42 `internal/ui`, 4 `internal/markup`, and 86 `internal/tui` files, their dependency edges, largest/hottest files, and recent rendering history. Existing uncommitted changes in `internal/app/app.go` and `internal/app/app_test.go` were not assessed or modified.

| 1 | HIGH | Regression / logical unit | `internal/ui/chatview.go:975-1024` | A header calls `flush()` (which appends the preceding or empty line), appends the visible `▾ title` to the new line, then stores `headerHit` on `lines[len(lines)-1]`—the preceding line. ## Quick wins

1. Add a red test for clicking the visible header marker, then fix the line association (finding #1).
2. Introduce and test a lossless `chatLine` offset/frame helper; migrate embed and component frames (finding #2).
3. Add narrow-width component tests before choosing wrapping versus ellipsis semantics (finding #4).
4. Make embedded field values use the existing markup renderer after the line helper is available (finding #5).
5. Decide whether picker thumbnails are a feature. Remove dead retained-tree/thumbnail code if not; otherwise add its draw path and one graphics test (finding #6).
6. Extract shared grapheme-aware `drawText` into `screen` (finding #9).
7. Consolidate duplicate header style resolution (finding #11).
8. Make link behavior honest: implement a safe open action or remove the clickable documentation/style (finding #3).
