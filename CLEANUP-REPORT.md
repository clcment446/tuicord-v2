# Code-base Cleanup Report — tuicord-v2 Go Discord client — 2026-07-14

## Verdict

The client is functionally healthy enough to build and test: `go test ./...` and `go vet ./...` pass when Go uses a writable cache, and the normalized store keeps Discord library types out of rendering code. The biggest problem is boundary growth around `internal/app`: it is a 1,266-line orchestration god object that wires every Arikawa capability, owns gateway registration, performs REST loading, and coordinates UI state. A full architectural rewrite is not warranted yet—the recent feature history shows broad cross-cutting work, but not several independent drivers duplicating one concept—though the Discord adapter boundary should be strengthened before the next major feature.

## Findings

| # | Severity | Check | Location | Problem | Fix |
|---|----------|-------|----------|---------|-----|
| 1 | HIGH | Keep files neat / clean logical units | `internal/app/app.go:1-1266` | `App` combines session construction, 20+ Arikawa capability interfaces, gateway subscription, event handlers, message actions, REST loaders, load guards, and UI callbacks. The file is over the 800-line MUST-FIX threshold and its `RegisterHandlers` function is a 140-line registration table. Every new Discord feature is likely to grow this hub. | Split by responsibility while keeping `package app`: `gateway_handlers.go` for registration and event adapters, `message_actions.go` for send/edit/delete/interactions, `directory_loaders.go` for guild/channel/role/history loading, and `app.go` for lifecycle/state wiring. Keep each file below ~400 lines and move repeated `ui.Post`/error completion logic into small named helpers. |
| 2 | HIGH | Split the layers / strict boundaries | `internal/app/app.go:21-26,103-218`; `internal/app/threads.go:10-13,182-225`; `internal/app/channels_manage.go`; `internal/app/roles.go` | The application layer directly imports Arikawa `api`, `discord`, `gateway`, `session`, and raw HTTP helpers. `New` exposes a concrete `*session.Session`, `App` stores one, and feature files build raw Discord REST payloads themselves. The `internal/discord` package therefore owns headers/session creation but not the client boundary. | Introduce an internal Discord port in `internal/discord` (or `internal/discord/client`): typed methods for directory, messages, threads, roles, channels, GIFs, and component/forum actions. Have one Arikawa-backed implementation translate to `store`/application DTOs. Make `app.New` depend on those ports and on a small gateway event source; hide `*session.Session` and `*api.Client` outside the adapter. Migrate one vertical slice at a time, starting with history and thread loading. |
| 3 | HIGH | Keep functions atomic | `internal/app/app.go:322-461` (`RegisterHandlers`), `internal/app/convert.go:369-520` (`convertComponentNode`) | `RegisterHandlers` is a long procedural list that also embeds event-specific preprocessing, while `convertComponentNode` is a 150-line type switch recursively converting every Components V2 variant. Both are difficult to review and each new event/component requires editing a central function. | Split handler registration into `registerMessageHandlers`, `registerGuildHandlers`, `registerMemberHandlers`, and `registerThreadHandlers`; each should only subscribe handlers. Split component conversion into per-family helpers (`convertContainerNode`, `convertInputNode`, `convertMediaNode`, etc.) behind the recursive dispatcher. Add focused table tests for each family. |
| 4 | HIGH | Honest documentation / concurrency | `internal/store/store.go:26-32`; `internal/app/threads.go:78-103` (especially line 87 in `loadArchivedThreads`) | The store explicitly says it is not safe for concurrent use and must only be accessed on the UI goroutine. `loadArchivedThreads` runs in a worker goroutine and reads the store to recover the parent guild before posting the result. This contradicts the concurrency contract and can race with UI mutations. | Capture the parent guild on the UI goroutine before starting the REST request, or post a closure that reads the parent and then starts a second worker. Add a regression test under `go test -race ./internal/app` that exercises archived-thread loading while a store mutation is queued. Update the package comment if the store contract is intentionally changed instead. |
| 5 | HIGH | Test hygiene / layer separation | `internal/discord/client_test.go:31-110,160-180` | Default tests call live Discord REST endpoints and fall back to reading `../../.env` for `TOKEN`. This makes the package dependent on network, account state, and a local secret file; it also explains the documented full-suite network failures. | Move live checks behind an `integration` build tag or a separate integration test package. Require an explicit `TOKEN` environment variable, never read `.env` from tests, and keep default tests on injected HTTP/gateway fakes. Add a small CI job that runs integration tests only when credentials are provisioned. |
| 6 | MEDIUM | Keep files neat | `internal/ui/chatview.go` (1,090 lines), `internal/ui/main.go` (833), `internal/app/convert.go` (700), `internal/store/store.go` (620) | These files each contain multiple cohesive domains: chat rendering/media/component interaction; navigation/sidebar/forum/composer state; all Discord-to-store conversions; and all store entities plus mutation/query logic. Large files increase merge conflicts and make feature ownership unclear. | Split `chatview.go` into rendering/media and interaction/component files; split `main.go` into sidebar/navigation and composer/persistence files; split `convert.go` into channel/message/component conversion files; split `store.go` into model definitions, message state, and guild/channel/member state. Preserve package-level APIs during the migration. |
| 7 | MEDIUM | Keep functions atomic / strict typing | `internal/discord/gif.go:22-41` | The GIF abstraction leaks the external `RequestJSON(any, ...)` shape into the package. The `any` destination is only used to make a single endpoint testable, so the seam is weaker than the domain operation it represents. | Replace `gifJSONRequester` with a typed `gifSearcherTransport` method such as `SearchGIFs(ctx, query) ([]GIFResult, error)`, or wrap Arikawa in an adapter that owns JSON decoding. Keep URL construction and response decoding in one tested adapter without exposing `any` to callers. |
| 8 | MEDIUM | Honest documentation / operational coupling | `internal/discord/build.go:12-20,29-69`; `internal/discord/client.go:40-62,84-125` | Session construction performs a live `https://discord.com/app` fetch through `sync.Once` to discover the build number, with a hard-coded fallback. A first session can block for 10 seconds, and one transient failure permanently memoizes the fallback for the process. The fallback's “reasonably current” claim cannot be verified by tests. | Inject an HTTP client and build-number resolver, test success/timeout/malformed-page cases, and make the startup policy explicit: bounded background refresh, a versioned configured fallback, or a clearly documented failure. Avoid hiding network I/O inside `identifyProperties`/`superProperties`. |
| 9 | LOW | Formatting / file hygiene | `internal/media/decode.go`, `internal/media/downscale_test.go`, `internal/store/color.go`, `internal/store/rich_test.go` | `gofmt -l` reports four first-party files. There are no remaining first-party TODO/FIXME markers, so this is the main mechanical hygiene issue. | Run `gofmt` on those four files and add formatting to the local verification command or CI. Do not mix that formatting-only change with the adapter refactor. |

## Quick wins

- Run `gofmt` on the four reported files.
- Add a `go test -race ./internal/app ./internal/discord` verification target.
- Add an integration build tag to `internal/discord/client_test.go` and remove its `.env` fallback.
- Extract `RegisterHandlers` into four registration helpers without changing behavior.
- Move `restForumPoster` and `restInteractionPoster` behind `internal/discord` ports.
- Add an injected build-number HTTP client so `fetchBuildNumber` can be tested without Discord.
- Add a package-level `internal/discord` documentation comment describing the REST/gateway boundary and browser-header policy.
- Run staticcheck in CI with a writable cache; the local run was blocked by the read-only global staticcheck cache.

## Follow-up status — 2026-07-14

The follow-up implementation resolved findings 1, 3, 4, 5, and 9 in the
working tree: gateway registration is split into focused helpers, archived
thread loading no longer reads the UI-only store from a worker, live Discord
tests are opt-in and offline defaults are deterministic, and all first-party
Go files are now gofmt-clean. The Discord adapter boundary and the remaining
large-file splits are still staged for a later vertical migration.

## Audit notes

- Scope: first-party `cmd`, `internal`, and `examples`; `third_party/arikawa` was treated as vendored code and excluded from cleanup findings.
- Repository inventory: 34,565 first-party Go lines; 204 first-party Go files; 69060 Go lines including vendored Arikawa.
- Recent-feature coupling evidence: commit `558bd34` touched 51 Go files for the rich embeds/forums/media change; the earlier forums/threads feature touched 27 files, picker 17, and sidebar ordering 14. These are broad feature slices, not evidence of seven duplicated procedural drivers, so no rewrite section is recommended yet.
- Verification: `go test ./...` passed. `go vet ./...` passed with `GOCACHE=/tmp/tuicord-go-build`; the default-cache vet run was blocked by the sandbox's read-only cache. `staticcheck` was attempted but could not write its global cache, so no staticcheck result is claimed. `gofmt -l` found four files.
